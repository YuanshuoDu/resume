### edict 三省六部架构（cft0808/edict, 13k+ stars）

**核心洞察**：1300年前的中国帝国制度，比现代多Agent框架更懂分权制衡。

**11个 Agent 分工**：
```
皇上（用户）
  ↓
太子（消息分拣：闲聊自动回复，旨意才建任务）
  ↓
中书省（规划：任务拆解和方案设计）
  ↓
门下省（审核：强制质量关卡，可封驳打回重做） ← 杀手锏
  ↓
尚书省（派发：根据类型分发给六部）
  ↓
六部 + 吏部（并行执行专项任务）
  ↓
奏折回报（完整审计链）
```

**为什么"门下省审核"是杀手锏**：
- CrewAI/AutoGen 的协作模式是"做完就交"——没有人检查产出质量
- 就像公司没有 QA，工程师写完代码直接上线
- 门下省强制在产出进入执行层之前过质量关
- **不受制约的权力必然会出错**——1300年前唐太宗就想明白了

**对陛下现有架构的启发**：
- 陛下现有的 "CTO → 前端/后端/QA" 链路缺少一个强制审核层
- 建议在 CTO 和子Agent之间加一个"门下省"角色
- 由 QA 工程师担任审核职能，但需要有**封驳权限**（打回重做），而不是只提交 Bug 列表

### 2026年趋势

- **A2A 协议**（Agent2Agent）成为互操作性标准（Google 推动）
- 60% 的 enterprise agentic AI pilot 在2026年失败——主要失败原因是架构治理问题
- **edict 的权限矩阵 + 状态机校验**是解决这个问题的关键

## 二、LaTeX microtype Typography 深入理解

### XeLaTeX vs LuaLaTeX 功能支持

| 特性 | XeLaTeX | LuaLaTeX |
|------|---------|----------|
| **protrusion (margin kerning)** | ✅ | ✅ |
| **expansion (font expansion)** | ❌ | ✅ |
| **tracking (letterspacing)** | ❌ | ✅ |
| **spacing (interword)** | ❌ | ✅ |

### 最佳实践配置（LuaLaTeX）

```latex
\usepackage[
  activate={true,nocompatibility},
  final,
  tracking=true,
  kerning=true,
  spacing=true,
  factor=1100,
  stretch=10,
  shrink=10
]{microtype}
```

参数说明：
- `factor=1100`： protrusion 量增加 10%（默认1000）
- `stretch=10, shrink=10`：减少行间弹性（默认20/20），避免行参差不齐
- `activate={true,nocompatibility}`：激活 protrusion 和 expansion

### 与 simpleresume.cls 的关系

**已知问题**（已在 LRN-20260326-002 记录并修复）：
- simpleresume.cls 设计时针对 pdflatex + microtype 全部特性
- CI 使用 XeLaTeX 时，expansion/tracking 不支持，导致错误
- 修复：XeLaTeX 下禁用 expansion 和 tracking

**残留问题**：
- 当前 CI 的 XeLaTeX 方案只能使用 protrusion，typography 质量有损失
- 如果要完整 microtype 特性，需要切换到 LuaLaTeX
- `lmodern` 字体在 XeLaTeX 下表现不佳（与 cls 设计不匹配）

## 三、对陛下项目的启发

### 1. 多 Agent 协作架构升级建议

**现状**：
```
CEO → CTO → 前端 / 后端 / QA
         （缺少强制审核层）
```

**建议改进**：
```
CEO → CTO → [门下省/QA 审核] → 前端 / 后端
              ↑ 
         有封驳权限，产出不达标打回
```

### 2. 简历 CI 优化方向

如果对简历排版要求极高，可以：
1. 评估切换到 LuaLaTeX（需要修改 CI 的编译命令）
2. 或者保持 XeLaTeX + protrusion-only，接受 typography 略逊
3. 当前迭代 crons 已经每周自动化优化三轮，暂时够用
